字体反爬(解密)

字体反爬是一种利用前端字体文件干扰文字正常显示，从而阻止爬虫直接获取正确文本内容的技术。它的核心思路是 “所见非所得” —— 用户在浏览器中看到的是正常的文字，但爬虫从HTML源码中获取的却是乱码或无关字符

cmap结构: 字体映射表

map数据信息 <map code="0x4e07" name="*#102"/>

  • code键名是十六进制
    • 是unicode表的排序位置
  • name="某个字形的标识符"
    • 表示是code字的在文件存储的位置名称
print(chr(0x4e07))
print(chr(0x4e09))
# 或 转换成十进制
a = eval("0x7F6E")
print(a)
c = chr(a)
print(c)

总结: 这个字在这个文件中是一个什么样的标识(name),

注意: cmap数据表中的数据结构是乱序的

GlyphOrder

存储的是字形的顺序和名称

  • id：字形在字体中的索引（位置）(与TTGlyph匹配)
  • name：字形的名称，用于在映射表中引用
<GlyphID id="108" name="*#102"/>

<!-- 在字体中，索引为108的位置上，存储着一个名为"*#102"的字形 -->

总结: 这个字的标识(name), 在这个文件中是在那个位置

无线debug

debugger 是JavaScript中定义的一个专门用于断点调试的关键字，只要遇到它，JavaScript的执行便会在此处中断，进入一种调试模式。有了 debugger 这个关键字，该 debugger 会被网站开发者利用，使其成为我们正常调试的拦截手段

案例演示

  • 目标网址：https://antispider8.scrape.center/
  • 进入该网站，开始我们爬虫正常的抓包控制台，你会发现弹出一个自动debug，如下图：

在我们没设置任何断点的情况下，它就直接进入了断点模式。这时候我们可以点击 Resume script execution(恢复脚本执行)按钮，尝试跳过这个断点继续执行，如下图所示：

然而不管我们点击多少次按钮，它仍然一次次地进入断点模式，无限循环下去，我们称这样的情况为无限debugger。那我们如何解决这种干扰手段呢？

解决手段

首先要做的是找到无限 debugger 的源头。在 Sources 面板中可以看到，debugger 关键字出现在一个 JavaScript 文件里，这时点击左下角的格式化按钮，那我们仔细观察该文件，如下图所示：

截图中setInterval方法表示循环，循环执行 debugger 语句。无限 for 循环、无限 while 循环也都能实现改需求，原理都大同小异

解决方法

禁用断点: 刷新后不会进入debugger,但是后续自身也无法使用断点

局部警用

在 debugger 语句所在的行数上单击鼠标右键，此时会出现一个快捷菜单，如下图所示：

操作完后会发现当前行数前多了一个？符号，同时 Breakpoints 也出现了刚才添加的断点位置，这时再次点击 Resume script execution(恢复脚本执行)按钮，就可以发现我们不会再进入无限 debugger 模式了

局部禁用

我们也可以选择另外一个选项 Add conditional breakpoint（添加条件判断）跳过debug，这个模式我们可以设置进入断点的条件,如下图所示： 会弹出一个输入框, 在里面输入false即可跳过

利用fiddler解除无限debug

  1. 找到对应的的debugger文件进行下载
  2. 下载本地后进行删除debug或修改,并保存,
  3. 在通过fiddler进行自动响应

记住: 浏览器一定要清除缓存, 浏览器有些网站会进行自动缓存

修改源代码解决

在Sources中选择Overriders点击 新建, 设置一个桌面的文件夹存储位置

  1. 在对应的js文件中,右键选择Override content
  2. 然后就可以在其中进行修改,删除debugger
