Vue风格指南

A类必须

vue推荐的必须项, 以公司规定为准

组件名

组件名应该始终由多个单词组成.除了根组件,以及transition, component之类的Vue内部组件

  • 推荐使用大驼峰 TodoPerfect 连接符单词 todo-perfect

    //组件名
name: 'TodoPerfect',

name: 'todo-perfect',

  • 这样做可以避免与现有以及未来的 HTML 元素产生冲突，因为所有的 HTML 元素名称都是单个单词的。

  • 避免组件重名几率

Props定义

应尽量详细,至少指定其类型

props: {
  status: String
}

v-for设置key

始终以 key 配合 v-for;在组件上必须始终以 key 配合 v-for，以便维护内部组件及其子树的状态

v-if 和 v-for

永远不要在一个元素上同时使用 v-if 和 v-for

  • 为了对列表中的项目进行过滤 显示 (比如 v-for="user in users" v-if="user.isActive")。在这种情形下，请将 users 替换为一个计算属性 (比如 activeUsers)，返回过滤后的列表, 不需要使用 v-if

    • 避免渲染列表
    • 解
    <ul>
  <template v-for="user in users" :key="user.id">
    <li v-if="user.isActive">   // 需要被隐藏的,和循环不会被冲突
      {{ user.name }}
    </li>
  </template>
</ul>

// data中数据
users:[
    {id:1,isActive:true},
    {id:2,isActive:true},
    {id:3,isActive:false},
    {id:4,isActive:true},
    {id:5,isActive:false},
    {id:6,isActive:true},
]

    <div v-for="user of usersss" :key="item.id">
    <p >{{item.id}}</p>
</div>

// 通过计算样式处理
computed:{
    usersss(){
        return this.arr.filter(item=>{
            return item.isActive
        })
    }
}

  • 避免渲染本应该被隐藏的列表 (比如 v-for="user in users" v-if="shouldShowUsers")。这种情形下，请将 v-if 移动至容器元素上 (比如 ul、ol)。

为组件样式设置作用域

对于应用来说，顶级 App 组件和布局组件中的样式可以是全局的，但是其它所有组件都应该是有作用域的

私有 property 名

个人感觉无用 后期在看,

B类强烈推荐

vue推荐的强烈推荐, 以公司规定为准

组件文件名大小写

单文件组件的文件名应该要么始终是单词大写开头 (PascalCase)大驼峰，要么始终是横线连接 (kebab-case)

components/
|- MyComponent.vue

components/
|- my-component.vue

组件多文件名系统化

只要有能够拼接文件的构建系统，就把每个组件单独分成文件 当你需要编辑一个组件或查阅一个组件的用法时，可以更快速的找到它。

components/
|- TodoList.vue
|- TodoItem.vue

紧密耦合的组件名

和父组件紧密耦合的子组件应该以父组件名作为前缀命名。

components/
|- TodoList.vue
|- TodoListItem.vue
|- TodoListItemButton.vue

components/
|- SearchSidebar.vue
|- SearchSidebarNavigation.vue

基础组件名

应用特定样式和约定的基础组件 (也就是展示类的、无逻辑的或无状态的组件) 应该全部以一个特定的前缀开头，比如 Base、App 或 V, 可以父类的组件名开头

components/
|- BaseButton.vue   谁的子类
|- BaseTable.vue
|- BaseIcon.vue

components/
|- AppButton.vue    app的子类
|- AppTable.vue
|- AppIcon.vue

components/
|- VButton.vue      v的子类
|- VTable.vue   
|- VIcon.vue

Props 名大小写

使用小驼峰

多个 attribute 的元素

多个 attribute 的元素应该分多行撰写，每个 attribute 一行

模板中简单的表达式

组件模板应该只包含简单的表达式，复杂的表达式则应该重构为计算属性或方法

attribute 值

非空 HTML attribute 值应该始终带引号 (单引号或双引号，以 JavaScript 中未使用的为准)

<input type="text">
<AppSidebar :style="{ width: sidebarWidth + 'px' }">

指令缩写

指令缩写 (用 : 表示 v-bind:、用 @ 表示 v-on: 和用 # 表示 v-slot:) 应该要么都用要么都不用

c类推荐

vue推荐的推荐, 以公司规定为准

组件/实例的选项的顺序

组件/实例的选项应该有统一的顺序。

  1. el 挂载
  2. name 组件名
  3. functional 函数式组件 没有响应式数据 查看文档
  4. delimiters 模板修改器 改变模板的编译方式; 比如插值符
  5. components 引人组件 模板内使用的资源
  6. mixins 混入 向选项里合并 property
  7. props 接收数据 组件的接口
  8. data 存储数据 本地的响应式 property
  9. computed 计算属性 本地的响应式 property
  10. watch 监听 通过响应式事件触发的回调
  11. 生命周期,按照生命周期执行顺序写
  12. methods 放置方法选项 不依赖响应系统的实例 property
  13. render 渲染页面 配合3 使用

组件找挂载地址 => 处理自身模板状态 => 引人外部组件 || 数据 => 处理自身数据 => 监听自身数据

=> 生命周期 => 方法 => 渲染页面

元素 attribute 的顺序

元素 (包括组件) 的 attribute 应该有统一的顺序

  1. is 定义 (提供组件的选项)
  2. v-for 列表渲染(创建多个变化的相同元素)
  3. v-if v-else-if v-else v-show v-cloak 条件渲染 (元素是否渲染/显示)
  4. v-pre v-once 渲染方式 (改变元素的渲染方式)
  5. ref key 唯一的 attribute (需要唯一值的 attribute)
  6. v-model 双向绑定(把绑定和事件结合起来)
  7. 其它 attribute (所有普通的绑定或未绑定的 attribute)
  8. v-on 事件 (组件事件监听器)
  9. v-html v-text 内容(覆写元素的内容)

组件渲染 => 循环渲染 => 单个标签渲染 => 渲染方式 => 唯一值 => 双向绑定数据 => 正常属性名 => 事件 => 覆写页面

D类谨慎使用

推荐不要这样去使用

scoped 中的元素选择器

元素选择器应该避免在 scoped 中出现

在 scoped 样式中，类选择器比元素选择器更好，因为大量使用元素选择器是很慢的

隐性的父子组件通信

应该优先通过 prop 和事件进行父子组件之间的通信，而不是 this.$parent 或变更 prop

  • 一个理想的 Vue 应用是 prop 向下传递，事件向上传递的。遵循这一约定会让你的组件更易于理解。然而，在一些边界情况下 prop 的变更或 this.$parent 能够简化两个深度耦合的组件。
  • 问题在于，这种做法在很多简单的场景下可能会更方便。但请当心，不要为了一时方便 (少写代码) 而牺牲数据流向的简洁性 (易于理解)。
